# SmartDNS Plus UI 待办

## P0: 查询追踪与决策审计

状态：待设计评审，暂不实现。

目标：让每一条查询记录能够解释 SmartDNS 为什么返回当前结果，而不仅是显示“查询发生过”。

### 目标体验

- 查询日志表格保持简洁，显示域名、类型、客户端、来源（缓存或上游）、分流组、结果摘要、耗时。
- 点击一条记录打开“查询详情”，展示完整决策过程。
- 历史记录没有追踪数据时明确提示“该记录生成于追踪功能启用前”。

### 详情应包含的数据

- 请求：域名、QTYPE、客户端、时间、命中规则与上游组。
- 缓存：是否命中缓存；命中时不虚构上游请求信息。
- 上游：参与查询的上游、协议、地址、响应耗时、成功或失败、是否 fallback。
- DNS Answer：A、AAAA、CNAME、TXT 等原始响应记录与 TTL。
- 选优：候选 IP 的测速结果、最终选中的 IP、选择依据。
- 最终状态：RCODE、拦截状态、总查询耗时。

### 采集与存储原则

- 以 SmartDNS 核心或插件回调提供的结构化事件为唯一可靠数据源。
- 不以解析 `smartdns.log` 或 `smartdns-audit.log` 作为正式实现；日志会轮转、字段不稳定，且无法可靠关联请求。
- 保留现有 `domain` 查询摘要表，避免影响仪表盘和普通查询日志性能。
- 新增短期追踪表，例如 `query_trace`、`query_trace_upstream`、`query_trace_answer`、`query_trace_candidate`。
- 摘要查询日志默认保留 30 天；详细追踪默认仅保留 24 小时或最近 10,000 条，并提供配置项。
- 只记录功能启用后的新查询；不尝试从旧日志回填历史结果。

### 实施阶段

1. 调研 SmartDNS 核心现有插件回调能否获得上游响应、Answer 与测速选优信息。
2. 若现有接口不足，设计最小的 SmartDNS 插件 API 扩展，并与核心版本兼容策略一并确定。
3. 为插件数据库增加追踪表和清理任务，完成数据库迁移。
4. 新增查询追踪 API：摘要列表、单条详情、保留策略配置。
5. 实现 Web UI 的“查询详情”抽屉；表格只显示精简摘要。
6. 使用缓存命中、分流、多个上游、fallback、A/AAAA/CNAME、失败响应等场景做回归测试。

### 验收条件

- 同一查询有多个上游或多个候选 IP 时，详情能完整且清晰地展示过程。
- 缓存命中、拦截、超时和 NXDOMAIN 不会错误显示为“已选中上游 IP”。
- 详情读取不会影响正常 DNS 查询性能。
- 追踪数据达到保留上限后能自动清理，数据库不会无限增长。
